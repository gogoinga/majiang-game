# H5 横屏紧凑化改版设计

日期：2026-04-28

## 背景

当前前端已经补过一轮手机横屏适配，但整体仍然保留较强的 PC 页面痕迹：

- 大厅页纵向留白偏大，房间输入与按钮排布更像普通 Web 表单
- 牌局页的 HTML 浮层尺寸偏大，和 Pixi 牌桌的紧凑横屏节奏不完全一致
- 页面大量尺寸仍然散落在 `px`、Tailwind 工具类和少量运行时缩放里，缺少统一基准

本次改版目标不是重写玩法或改动牌桌核心逻辑，而是在保持现有功能和 Pixi 桌面结构的前提下，把大厅页和牌局页统一成一套 H5 优先、横屏优先、紧凑型的视觉体系。

## 已确认决策

- 目标方向：`H5 优先但兼容 PC`
- 视觉密度：`紧凑型`
- 覆盖范围：`大厅页 + 牌局页`
- 推荐方案：`B. 紧凑 H5 版`

## 目标

1. 手机横屏进入时，整体观感更像手游桌面，而不是 PC 页面压缩版。
2. PC 上继续复用同一套布局结构，只在尺度上更宽松，不维护第二套桌面布局。
3. 引入统一的 `rem` 尺寸体系，收敛按钮、弹窗、标题、边距、状态条和卡片的尺寸来源。
4. 保持现有房间功能、对局流程、Socket 事件和 Pixi 牌桌渲染逻辑可用。

## 非目标

- 不改麻将规则、计分规则、Socket 协议或后端状态机
- 不重做 Pixi 牌桌为纯 DOM 布局
- 不新增新的大厅功能或新的游戏交互
- 不在本次改版中重构所有历史文件，只覆盖大厅页、牌局页和必要的全局尺寸入口

## 设计总览

整体方案采用“双层适配”：

- DOM / HTML 浮层层：使用 `rem` 作为主尺寸单位，统一大厅页与牌局页的视觉节奏
- Pixi 牌桌层：继续使用视口尺寸和运行时比例布局，但进一步收紧手机横屏参数，使其与新的 `rem` 外层 UI 协调

这样可以在不破坏现有牌桌代码结构的前提下，拿到更稳定的 H5 横屏视觉结果。

## 尺寸体系

### 根字号策略

在全局样式里引入动态根字号，基于“横屏短边优先”的思路计算：

- 手机横屏：根字号跟随短边变化，保证小屏幕也能获得一致的紧凑密度
- PC：根字号继续随视口变化，但设置上限，避免控件被放大得过头

建议策略：

- 使用 CSS 变量承接根字号计算结果
- 通过 `clamp()` 设置根字号下限和上限
- 为常用间距、圆角、阴影、安全边距建立全局变量，例如：
  - `--app-shell-padding`
  - `--panel-radius`
  - `--control-height`
  - `--compact-gap`
  - `--safe-bottom`

### 单位收口

大厅页与牌局页中的以下元素优先改为 `rem`：

- 标题字号
- 按钮高度与横向内边距
- 弹窗宽度、圆角、内边距
- 房间卡片边距和信息间隔
- 计分角标尺寸
- 返回大厅按钮
- 底部状态栏高度
- 倒计时提示框
- 碰/杠/胡/过等操作按钮

Pixi 层保留按视口计算，但会减少和 DOM 浮层之间的尺度冲突。

## 页面布局设计

### 1. 房间大厅页

大厅页改成更接近游戏大厅的紧凑横屏结构：

- 顶部 Hero 压缩成低高度横向头区，保留标题和一句说明，但明显减少纵向占用
- `roomId 输入框 + 加入房间 + 创建房间 + 刷新列表` 作为一组紧凑控件，优先保证横屏下一屏可读
- 房间列表卡片改为更密的网格布局，单卡只保留关键数据：
  - `roomId`
  - 状态
  - 在线人数
  - 准备人数
  - 进入按钮
- 小屏横屏下允许控件换行，但保持统一高度和节奏，不出现按钮挤压换成两行文字的情况

### 2. 牌局页

牌局页继续以 Pixi 桌面为中心，但外层浮层全面改成紧凑 H5 节奏：

- 返回大厅按钮：缩成轻量悬浮胶囊按钮，贴近左上角
- 昵称弹窗 / 局结束弹窗：统一宽度、圆角、内边距和字号比例
- 底部准备栏：改成更矮的横向状态条，避免未开始时过度占高
- 倒计时浮层：减少尺寸，突出数字本身
- 碰/杠/胡/过按钮：统一按钮高度和圆角，底部操作区更贴近手机横屏游戏
- 计分角标：进一步紧凑化，保持贴座位定位但减少体积

### 3. 横屏根容器

根容器负责建立统一的 H5 壳子：

- 横屏时页面以“满屏游戏容器”方式展示
- 继续保留竖屏遮罩和“点击尝试横屏”
- 统一滚动策略：
  - 大厅页允许纵向滚动
  - 牌局页保持全屏固定舞台
- 处理移动端安全区域边距，避免贴边元素在部分设备上过近

## 实施原则

1. 先统一尺寸体系，再改局部页面细节，避免每个组件各自为战。
2. 先改大厅页和牌局页的 DOM 壳层，再微调 Pixi 参数，降低联动风险。
3. 保持功能事件和状态逻辑不动，避免布局改造引入玩法回归。
4. 任何改动都优先兼容手机横屏，其次让 PC 作为同布局放大版自然成立。

## 风险与应对

### 风险 1：DOM 浮层缩小后与 Pixi 牌桌发生遮挡

应对：

- 保留关键浮层的贴边策略
- 在 `TableScene` 中微调牌桌边缘安全区
- 优先保证操作按钮和弹窗不压到手牌

### 风险 2：rem 根字号过于激进，导致 PC 显示过大或手机过小

应对：

- 使用 `clamp()` 控制上下限
- 通过少量 CSS 变量统一修正，而不是到处打补丁

### 风险 3：大厅页横屏下按钮与输入区再次挤压换行

应对：

- 输入框、按钮统一最小高度
- 允许布局级换行，不允许文本级换行
- 按钮文案保持短词策略

## 验收标准

1. 手机横屏下，大厅页首屏能稳定显示：
   - 标题区
   - 快速加入/创建控件
   - 至少一排房间卡片
2. 牌局页横屏下，Pixi 牌桌保持完整可见，关键浮层不明显遮挡手牌和主要牌河。
3. 返回大厅、准备、碰/杠/胡/过、继续等按钮在横屏下不出现文字换行。
4. 昵称弹窗、结算弹窗、倒计时框、计分角标整体尺寸明显比当前更紧凑。
5. PC 打开时继续使用同一套布局结构，不出现严重拉伸或信息拥堵。

## 验证计划

- 执行 `pnpm --dir web-ui build-only`
- 手动验证大厅页：
  - 小屏横屏宽度
  - 普通 PC 宽度
  - 房间列表可滚动
  - 输入与按钮不换行或挤爆
- 手动验证牌局页：
  - 未开始状态栏
  - 对局中倒计时
  - 操作按钮区
  - 昵称弹窗和结算弹窗
  - 横竖屏切换后的重排

## 实施后预期结果

完成后，前端会形成一套统一的 H5 横屏视觉框架：

- 大厅页更像游戏大厅
- 牌局页更像移动端麻将桌面
- DOM 浮层和 Pixi 牌桌在尺度上更一致
- 后续再调细节时，可以直接基于 `rem` 变量而不是继续堆积零散 `px`
